我玩了一个星期的react的理解

Posted on 2020-05-21 In 前端 Views: Valine:

不知不觉，我居然已经玩了一个星期的react了，因为项目需要，所以一切从0开始，如大家所了解的那样，react上手的的确确要比vue难一些，但是也并没有传说中的那么难。按照我的套路，我感觉上手react会非常容易。

1、首先，我们应该了解一下，一个简单的react应用是怎么样的。

我们先不说啥，看到这个结构，实际上和Vue是很像的，并没有太大的区别，从大的模式上来看，就是将Virtual Dom，渲染到一个真实的Dom节点上去。所以，这一点上和Vue似乎并不是太大。直接就理解过来了，因此，我们理解，React实现一个web页面，实际上就是很多个React.Component打积木即可。需要注意的是，原生Dom节点上如果需要使用class属性，是需要使用className的，因为class是jsx的一个关键字。这点略有点不爽，不能直接复制重构给到的html结构，不过好在webstorm会直接帮我们转换。

关于生命周期，其实和Vue并没啥大的区别 ，这里都懒得啰嗦了。需要注意的是，componentDidMount() 方法会在组件已经被渲染到 DOM 中后运行，因此比较适合做网络请求等，在 componentWillUnmount() 生命周期方法中取释放一些资源已防止内存泄漏等，然后根据 shouldComponentUpdate() 的返回值，判断 React 组件的输出是否受当前 state 或 props 更改的影响，因此你可以在这里做一些渲染优化相关。

2、函数式组件

为啥会出现一个函数式组件，是因为，类组件给人的感觉比较重，写一个函数，明显就显得轻量一点，然后，函数式组件因为不存在生命周期钩子，所以，出现了React Hooks这样的工具，从而使得函数式组件能够玩出类组件的效果，而且代码还比类组件简洁，因此，建议优先考虑使用函数式写组件。

3、高阶组件HOC

高阶组件，听起来名字就吓老子一跳，因为10年前上大学的时候，高数中有一个高阶函数，听起来就比较吓人，但是实际上，高阶组件其实就是对组件进行了一个封装，将一些公共的功能分装起来，是的被封装的组件具备一类功能而已，或者说，为封装组件过滤一些props，避免敏感信息暴露等作用。

4、React Hooks

这玩意就是为函数组件应运而生的，对于刚吃React 瓜的人，建议先玩一玩 useState,useEffect 这两个hook。本来函数组件是没有状态的，但是似乎用上这两个hook 之后，组件变得可以为之状态了。但是不可以大以轻心，因为useEffect可能会存在坑，useEffect可以在里面做事件监听，但是不要忘记在返回函数中注销监听事件，不然内存泄漏就找上你了。

4、Redux、React-redux

可以说，玩React如果没有玩过Redux，那说明你还没有玩React，如果你只写了一个Hello world，请出门右拐，为什么会有Redux，是以为有管理全局状态的需求，试着想一下，加入有AB两个子组件，平行的关系，非父子关系，需要共享数据，且可以修改数据，那么这时候Redux就闪亮登场了。但是Redux实际上不是为React单独设计的，因此如果需要在React中使用Redux的化，需要引入React-redux，被吓到了吧，别害怕，这个东西就是提供了一些封装函数将redux的功能赋能给你的组件而已，把你的组件包装成一个高阶组件。无非就是提供了一些工具函数，比如mapStateToProps,mapDispatchToProps，把全局状态转换为你组件的state，把dispatch方法转换为你props中的方法。

你以为你这是只开始，react作为状态管理，dispatch方法实际上只是支持同步的，那如何使用异步的dispatch呢，如是redux-thunk,redux-sage等出现了。

你以为结束了吗，没这又是一个开始，每次state变化会导致mapStateToProps引导新的props，数据没变化，但是页面重新渲染了，这不科学啊，如是reselectc出现了，我还真不知道后面会出现哪些花里胡哨的东西了。

5、服务端渲染-next框架

你以为就这样结束了吗，并没有，ssr了解过了吗，next框架拿过去想用，一个比较优秀的react 服务端渲染框架，对于服务端渲染，没有什么好说的，之前php，嗯，那个世界上最好的语言做web的时候，实际上就是将整个页面渲染出来，更本就不想现在开始渲染出来的知识一个页面骨架，然后，js异步请求数据，数据回来，在更新dom，这样对于有些搜索引擎来说，并不友好啊，拿不到你的首屏内容，如是装逼的服务器渲染回来了，一个个活生生的浏览器端渲染，如何做到服务端渲染呢。

大的方向分三步走：

1、起一个node server

2、用户访问一个页的时候，发现这个页需要异步加载数据，那好，node server直接将这些数据在服务端加载好，然后注入到页面中一并返回，通常是JSON.stringf…

3、客户端发现 已经有数据了，那感情好啊，直接拿来用就是啦，不用哥在请求数据了。

以上过程叫做注水，脱水,听起来名字挺高大上的，但实际上就是那么回事，里面的难点就是要做同构，同构又是他妈什么意思，同构其实是说，一套代码既能够在服务端跑，也能够在客户端跑，所以，window这些东西，就麻烦别放到服务端可能走到的逻辑中去了，还记得我们的生命周期componentDidMount()，嗯dom加载之后，所以，大哥，你在这里用dom是安全的。
